# 设计说明：shop-update

## 概览
本次改造围绕两条主线展开：

1. 将商户详情缓存从“固定 TTL 的缓存旁路”升级为“逻辑过期 + 异步重建 + 空值缓存”的热点缓存模型。
2. 为商户补充新增、更新、删除接口，并在写路径中统一维护数据库、详情缓存与 GEO 索引一致性。

---

## 当前状态
### 已有能力
- `ShopController.java` 提供商户列表、商户详情、附近商户与商户类型查询接口。
- `ShopServiceImpl.java` 使用 `CacheClient.queryWithPassThrough()` 查询商户详情，并通过空值占位防护缓存穿透。
- `queryNearby()` 已基于 `shop:geo:{typeId}` 读取 Redis GEO 索引，按距离分页返回商户。

### 现存问题
- 商户详情缓存使用固定 TTL，热点商户在过期瞬间会同步回源数据库，存在缓存击穿风险。
- 多个商户详情键可能在相近时间过期，存在缓存雪崩风险。
- 当前没有商户新增、更新、删除接口，也没有缓存/GEO 写路径维护逻辑。
- 一旦后续补写操作而不设计缓存一致性，详情与附近商户数据会出现脏读或漏删。

---

## 目标架构

```text
读商户详情 /shops/{id}
   ↓
CacheClient.queryWithLogicalExpire()
   ├─ 命中未过期缓存 -> 直接返回
   ├─ 命中过期缓存 -> 返回旧值，并尝试异步重建
   ├─ 命中空值占位 -> 返回不存在
   └─ 未命中 -> 回源 DB，写入逻辑过期缓存或空值缓存

写商户
   ├─ 新增 POST /shops
   │    ├─ insert DB
   │    ├─ add GEO
   │    └─ delete detail cache
   ├─ 更新 PUT /shops/{id}
   │    ├─ read old shop
   │    ├─ update DB
   │    ├─ update GEO if typeId/x/y changed
   │    └─ delete detail cache
   └─ 删除 DELETE /shops/{id}
        ├─ read old shop
        ├─ delete DB
        ├─ remove GEO
        └─ delete detail cache
```

---

## 详细设计

## 1. 商户详情逻辑过期缓存模型
### 方案
新增逻辑过期缓存结构，例如 `RedisData`：
- `data`：真实商户详情对象
- `expireTime`：逻辑过期时间

`CacheClient` 增加类似 `queryWithLogicalExpire()` 的能力，负责：
- 读取字符串缓存并区分三种状态：真实缓存、空值占位、缓存未命中。
- 命中未过期逻辑缓存时直接返回。
- 命中过期逻辑缓存时，先返回旧值，再尝试获取 `lock:shop:{id}` 互斥锁触发后台重建。
- 未命中时同步回源数据库并写入新的逻辑过期缓存。

### 设计要点
- 逻辑过期只应用于商户详情，不扩大到商户列表或附近商户列表。
- 继续保留 `NULL_PLACEHOLDER`，不存在商户仍用短 TTL 空值缓存。
- 热点缓存重建只允许一个线程进行，其余线程继续读旧值，避免击穿。
- 逻辑过期时间仍以 30 分钟为基准，但写入时增加随机抖动，降低同窗失效带来的雪崩风险。

### 原因
商户详情属于高频公开读场景，允许极短时间返回旧值来换取高并发下的稳定性，比固定 TTL 回源更适合热点数据。

---

## 2. 互斥锁与异步重建
### 锁设计
新增常量：
- `LOCK_SHOP_KEY = "lock:shop:"`

锁粒度采用 `shopId` 级别，确保只有同一个热点商户的重建互斥，不影响其他商户。

### 重建方式
`CacheClient` 内部维护轻量线程池，拿到锁的线程负责：
1. 回源数据库查询商户。
2. 如果存在，写入新的逻辑过期缓存。
3. 如果不存在，写入空值缓存。
4. 释放锁。

### 返回策略
- 命中过期缓存：当前请求直接返回旧值。
- 没拿到锁的请求：不阻塞等待，直接返回旧值。
- 完全未命中：同步回源并写入逻辑过期缓存。

### 原因
逻辑过期的核心价值是把“过期重建”从主请求路径中剥离出去，避免热点 key 在过期瞬间把流量全部打向数据库。

---

## 3. 缓存雪崩缓解
### 方案
即使采用逻辑过期，也应对商户详情缓存引入 TTL 抖动：
- Redis 物理 TTL 仍保留一个较长兜底值，例如大于逻辑过期时间。
- 逻辑过期时间或物理 TTL 增加随机偏移，避免大量商户在同一时间集中进入重建窗口。

### 原因
逻辑过期降低的是热点击穿，但如果大量键都在同一时刻进入重建窗口，数据库仍可能遭受集中回源。抖动能进一步分散压力峰值。

---

## 4. 商户新增接口设计
### 接口
- `POST /shops`

### 输入
使用 `Shop` 或专用 DTO 承载商户基础字段：
- `name`
- `typeId`
- `address`
- `x`
- `y`
- `images`
- `avgPrice`
- `comments`
- `score`
- `openHours`

### 处理步骤
1. 校验必要字段。
2. 向 `tb_shop` 插入商户记录。
3. 以 `typeId + 坐标 + shopId` 写入 `shop:geo:{typeId}`。
4. 删除 `cache:shop:{id}`（即使理论上大概率不存在，也保持写路径一致）。

### 返回建议
返回新建后的商户详情或新商户 ID。

---

## 5. 商户更新接口设计
### 接口
- `PUT /shops/{id}`

### 处理步骤
1. 先查询旧商户记录，不存在则返回业务异常。
2. 更新数据库记录。
3. 比较旧值与新值：
   - `typeId` 变化：从旧 `shop:geo:{oldTypeId}` 删除，再加入新 `shop:geo:{newTypeId}`。
   - `x/y` 变化：更新 GEO 坐标。
   - 都未变化：GEO 不动。
4. 删除 `cache:shop:{id}`，让下次详情查询重建逻辑过期缓存。

### 为什么更新后删除缓存
用户已明确要求“增/更新后也删除缓存”。当前项目也更适合 cache-aside 的写后失效策略，避免写路径直接构建复杂缓存结构。

---

## 6. 商户删除接口设计
### 接口
- `DELETE /shops/{id}`

### 处理步骤
1. 先查询旧商户记录，不存在则返回业务异常。
2. 从数据库物理删除商户。
3. 从 `shop:geo:{typeId}` 删除该商户的 GEO 成员。
4. 删除 `cache:shop:{id}`。

### 删除语义
第一版采用物理删除。

### 原因
当前实体没有逻辑删除字段，物理删除更符合现状，也能让详情缓存和 GEO 清理路径保持简单直接。

---

## 7. 服务接口与职责调整
### `ShopService`
在现有查询接口基础上新增：
- `Shop createShop(Shop shop)`
- `Shop updateShop(Long shopId, Shop shop)`
- `void deleteShop(Long shopId)`

### `ShopController`
新增对应写接口：
- `POST /shops`
- `PUT /shops/{shopId}`
- `DELETE /shops/{shopId}`

### `ShopServiceImpl`
继续承担：
- 商户详情缓存查询
- 商户写操作事务编排
- 详情缓存删除
- GEO 索引维护

---

## 8. GEO 一致性维护
### 新增
在 `shop:geo:{typeId}` 中执行 `GEOADD`。

### 更新
根据旧值/新值差异执行：
- 旧类型删除
- 新类型新增
- 或原类型坐标覆盖写入

### 删除
执行 `ZREM/GEO remove` 删除成员。

### 原因
附近商户查询完全依赖 GEO 索引，如果写路径不维护 GEO，附近商户功能会直接出现错位、漏查或脏数据。

---

## 9. 一致性与异常处理
### 一致性原则
- 以 MySQL 为最终真值。
- 写操作先成功落库，再维护 Redis GEO 与详情缓存。
- 详情缓存统一采用“写后删除、读时重建”。

### 异常处理
- 数据库写失败：直接回滚，不触碰 Redis。
- Redis/GEO 删除失败：记录中文错误日志，便于人工修复或后续补偿。

### 权衡
当前阶段优先保持主链路清晰，不引入 MQ/订阅式缓存同步等更复杂机制。

---

## 10. 测试与验证建议
### 单元/集成测试重点
1. 商户详情缓存：
   - 命中未过期缓存
   - 命中过期缓存并触发异步重建
   - 未命中同步回源
   - 空值缓存命中
2. 商户新增：
   - DB 新增成功
   - GEO 写入成功
   - 详情缓存删除调用发生
3. 商户更新：
   - 普通字段更新
   - `typeId` 变化导致 GEO 搬迁
   - `x/y` 变化导致 GEO 坐标更新
   - 详情缓存删除调用发生
4. 商户删除：
   - DB 删除成功
   - GEO 删除成功
   - 详情缓存删除成功

### 人工验收主路径
1. 连续请求同一商户详情，验证逻辑过期窗口内仍能快速返回。
2. 修改商户名称或坐标后，再次访问详情与附近商户，验证新数据已生效。
3. 删除商户后，详情查询返回不存在，附近商户中不再出现该商户。

## 11. 决策汇总
| 主题 | 选型 |
|---|---|
| 详情缓存模型 | 逻辑过期 + 异步重建 |
| 穿透防护 | 保留空值缓存 |
| 击穿防护 | `shopId` 粒度互斥锁 |
| 雪崩缓解 | 逻辑过期/物理 TTL 抖动 |
| 写后缓存策略 | 新增/更新后删除缓存，删除后删除缓存 |
| 删除语义 | 物理删除 |
| GEO 维护 | 写路径同步维护 |
| 最终真值 | MySQL |
| 缓存一致性策略 | 写后失效，读时重建 |
| 风险取舍 | 允许短时间返回旧值，换取热点稳定性 |

## 12. 关键 Redis Key 与排查说明
### 关键 Key
- `cache:shop:{id}`：商户详情热点缓存。正常情况下保存 `RedisData(data, expireTime)`；不存在商户时保存空值占位。
- `lock:shop:{id}`：商户详情热点重建锁。仅在逻辑过期后异步重建期间短暂存在。
- `shop:geo:{typeId}`：按商户类型分片保存 GEO 索引，用于附近商户查询。

### 典型排查路径
#### 场景 1：商户详情返回旧数据
1. 检查 `cache:shop:{id}` 是否仍存在旧值。
2. 检查逻辑过期时间是否已经到期但尚未完成异步重建。
3. 查看应用日志中是否出现“热点缓存异步重建失败”或“热点缓存重建锁竞争中”日志。
4. 如确认缓存异常，可删除对应 `cache:shop:{id}`，再触发一次详情查询重建。

#### 场景 2：附近商户结果与详情不一致
1. 检查商户数据库中的 `typeId/x/y` 是否已更新成功。
2. 检查对应 `shop:geo:{typeId}` 是否仍保留旧成员，或新类型 GEO 中缺少该商户。
3. 查看应用日志中是否出现“商户GEO类型索引已迁移”“商户GEO坐标已更新”“商户GEO索引删除完成”等日志。
4. 如 GEO 索引异常，可按当前数据库真值重新写入对应 GEO 成员。

#### 场景 3：新增/更新后详情未刷新
1. 检查写请求是否成功执行到缓存删除步骤。
2. 检查日志中是否出现“商户详情缓存删除完成”或“商户新增/更新后的缓存或GEO维护失败”。
3. 如果数据库已成功但 Redis 删除失败，应优先删除对应 `cache:shop:{id}`，再重新访问详情接口。

## 13. 联调说明
### 后端接口
- `GET /shops/{shopId}`：查询商户详情，匿名可访问。
- `POST /shops`：新增商户。
- `PUT /shops/{shopId}`：更新商户。
- `DELETE /shops/{shopId}`：删除商户。
- `GET /shops/nearby`：按类型和坐标查询附近商户。

### 调用方注意事项
- 当前变更只补后端接口，不包含后台管理前端页面。
- 新增/更新接口提交后，详情缓存会被删除，下一次详情查询会按最新数据库值重建逻辑过期缓存。
- 如果更新了 `typeId/x/y`，附近商户结果依赖 GEO 维护成功，联调时应同时验证详情与附近商户两条链路。
- 删除商户后，调用方应预期详情接口返回“商户不存在”，且附近商户结果不再出现该商户。

### 当前结论
本次无需同步修改现有用户前台页面；如后续引入后台管理端，可直接基于上述商户增删改接口联调。
