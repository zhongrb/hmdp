# 变更提案：shop-update

## 背景
当前商户详情查询已经具备基础缓存旁路与空值占位能力，但仍存在两个明显问题：
1. 详情缓存采用固定 TTL，热点商户在缓存过期后会同步回源数据库，存在缓存击穿风险。
2. 多个商户详情键如果在相近时间失效，容易集中回源数据库，存在缓存雪崩风险。

与此同时，当前项目只提供商户查询接口，尚未补齐商户新增、更新、删除能力。一旦后续补充商户写操作，如果没有同步设计缓存与 GEO 索引维护逻辑，就会出现详情缓存脏读、附近商户结果错误或缓存/GEO 遗漏清理的问题。

## 变更目标
1. 将商户详情缓存升级为“逻辑过期 + 异步重建”的热点缓存模型。
2. 保留空值缓存能力，继续防护缓存穿透。
3. 新增商户增删改接口与服务逻辑，覆盖新增、更新、删除三条写路径。
4. 在商户新增、更新后删除详情缓存，使下次查询按最新数据重建；删除商户后同步删除详情缓存与 GEO 索引。
5. 当商户类型或经纬度变化时，正确维护 Redis GEO 索引，保证附近商户查询结果一致。

## 用户价值
- 热点商户详情在缓存过期窗口内仍能稳定返回，降低高并发下数据库抖动风险。
- 商户后台或运营写入后，详情查询与附近商户能力能够更快反映最新状态。
- 统一建立商户数据的读写一致性规则，方便后续扩展后台管理能力。

## 变更范围
### 包含
- `ShopService` / `ShopServiceImpl` 的详情缓存模型升级。
- `CacheClient` 对逻辑过期、异步重建、空值缓存共存的支持。
- 商户新增、更新、删除接口与服务实现。
- 商户写操作后的详情缓存删除策略。
- 商户 GEO 索引的新增、更新、删除维护。
- 对热点缓存与商户写路径补充测试。

### 不包含
- 独立后台管理前端页面。
- 商户列表分页缓存。
- 商户批量导入/批量删除接口。
- 复杂的多级缓存、消息补偿或订阅式缓存同步方案。

## 风险与权衡
- 逻辑过期会允许短时间返回旧值，读一致性不是强实时，但可以换取热点读链路稳定性。
- 新增写接口后，需要额外维护 GEO 索引与缓存删除逻辑，否则比单纯查详情更容易出现数据不一致。
- 写后删除缓存会让下一次详情查询冷启动一次，但实现更稳妥，也更符合当前项目的 cache-aside 风格。

## 成功标准
- 商户详情热点查询命中过期缓存时，能够返回旧值并触发后台重建，而不是所有请求同时打到数据库。
- 新增、更新、删除商户后，详情缓存与 GEO 索引都能按规则同步维护。
- 商户新增/更新后详情缓存会被删除，删除商户后详情缓存和 GEO 记录会被移除。
- 匿名商户详情查询、附近商户查询、商户增删改三类主路径均可验证通过。
