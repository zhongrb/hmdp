# 变更提案：voucher-update

## 背景
当前优惠券秒杀链路采用 Redis Lua 预校验后同步落库的方式，能够满足基础抢券需求，但在高并发下仍存在主请求线程承担数据库写入压力、响应耗时受限、订单号依赖数据库插入结果等问题。

## 变更目标
1. 将秒杀抢券链路升级为“Redis 资格裁决 + RabbitMQ 异步落库”。
2. 订单 ID 改为雪花算法生成的全局唯一 Long 值，在抢券入口即时返回给前端。

## 用户价值
- 抢券接口只承担资格校验与消息投递，响应更快。
- 异步消费削峰，降低高并发下数据库瞬时写入压力。
- 订单号可在前端即时展示，便于后续查询与排障。

## 变更范围
### 包含
- 雪花算法订单 ID 生成器。
- RabbitMQ 秒杀下单消息投递、消费者处理、ACK 与死信队列基础配置。
- Lua 脚本调整：继续负责库存和一人一单原子校验。
- 数据库扣库存改为原子更新语句，订单落库改为异步消费执行。

### 不包含
- 独立后台前端页面。
- 优惠券订单详情页或轮询查询页。
- 复杂的本地消息表 / Outbox 事务投递方案。
- 跨实例横向扩容下的高级消费编排优化。

## 风险与权衡
- RabbitMQ 无法像 Redis Stream 一样由 Lua 原子完成入队，存在“Redis 预扣成功但 MQ 发送失败”的窗口，需要先通过日志、确认机制与后续补偿策略降低风险。
- 雪花算法需要明确机器位与时间位配置，避免时钟回拨影响唯一性。

## 成功标准
- 抢券接口在 Lua 校验成功后返回雪花算法订单 ID，不再同步创建数据库订单。
- RabbitMQ 消费者能够完成数据库扣库存与订单创建，并通过手动 ACK 确认消息。
- 仍然保持“一人一券”“库存不超卖”的业务约束。
